## Workflow: Adding a New Case Study

### Step 1: Prepare Editorial Content

Use the **[Product Design Case Study Skill](./codex-skills/product-design-case-study/SKILL.md)**.

Start with either:

- **source-led**: Existing Markdown, notes, or partial structure
- **image-first**: Primarily screenshots; text needs reconstruction

Before writing:

1. Read `cases/agent-layout-guidelines.md` for layout constraints
2. Check `codex-skills/product-design-case-study/references/case-schema.md` for mandatory sections
3. Run `node codex-skills/product-design-case-study/scripts/inventory_case_assets.mjs <markdown> <image-folder>` to audit visuals

Output: Markdown file in `cases/<slug>.md` + image map in `cases/<slug>-image-map.md`

### Step 2: Convert to JSX Component

Read [cases/agent-layout-guidelines.md](./cases/agent-layout-guidelines.md) for:

- Base component structure
- Typography classes (headings, paragraphs, emphasis)
- Table, card, and list patterns
- Image/Mermaid diagram inclusion

Create `src/app/data/<slug>Content.tsx` exporting a component:

```tsx
export function MyProjectContent() {
  return (
    <div
      className="max-w-4xl space-y-20 text-neutral-400"
      style={{ fontSize: '0.95rem', lineHeight: 1.8 }}
    >
      {/* sections */}
    </div>
  );
}
```

### Step 3: Register Project & Route

1. **Extend `ProjectCaseStudy` type** in `src/app/data/projects.ts` if slug is new:

   ```ts
   export type ProjectCaseStudy = 'monetix' | 'unimedpay' | 'mynewslug';
   ```

2. **Add entry to `projects` array**:

   ```ts
   {
     slug: 'mynewslug',
     title: 'My Project',
     caseStudy: 'mynewslug',
     // ... other fields
   }
   ```

3. **Register lazy import in `src/app/components/CaseStudyPage.tsx`**:
   ```ts
   const caseStudyContentMap = {
     monetix: lazy(() =>
       import('../data/monetixContent').then((m) => ({ default: m.MonetixContent }))
     ),
     unimedpay: lazy(() =>
       import('../data/unimedpayContent').then((m) => ({ default: m.UnimedPayContent }))
     ),
     mynewslug: lazy(() =>
       import('../data/mynewslugrContent').then((m) => ({ default: m.MyProjectContent }))
     ),
   };
   ```

### Step 4: Verify & Deploy

```bash
npm run build          # Verify no build errors
npm run validate       # lint + type-check
npm run test          # Ensure no regressions
```

## Workflow: Fixing TypeScript Errors

1. Run `npm run type-check` to see all TS errors
2. For import/module issues:
   - Ensure path alias `@` is only used in tests (vitest.config.ts)
   - Use relative imports in app code (`import { X } from '../sibling'`)
3. For styled-component issues:
   - Never use inline styles; convert to Tailwind classes
   - Only exception: `ScrollZoomImage.tsx` (dynamic CSS transforms)
4. For missing types:
   - Check if component/data file is exporting correctly
   - Verify extension matches (`.ts` vs `.tsx`)

---

## Workflow: Fixing ESLint or Import Sort Errors

1. Run `npm run lint` to see violations
2. Auto-fix what you can:
   ```bash
   npm run lint:fix && npm run format
   ```
3. If import sort fails:
   - `eslint-plugin-simple-import-sort` enforces a strict order (error level)
   - Order: React/third-party → relative imports → blank line
   - Never use `/* eslint-disable-next-line */` without discussing with maintainer

## Workflow: Adding a New Route

1. Define route in `src/app/routes.tsx`:

   ```tsx
   { path: '/mypage', component: MyPage }
   ```

2. Create component in `src/app/components/MyPage.tsx`:
   - Use `useDocumentTitle()` hook to set page title
   - Import institutional copy from `src/app/data/site.ts` if needed

3. Add navigation link in `src/app/components/Header.tsx` if public

4. Test with `npm run dev` (localhost:5173)

## Workflow: Updating Institutional Copy

All user-facing text (hero, positioning, contact, practice, social) lives in `src/app/data/site.ts`:

1. Edit the relevant field in `siteConfig` or `portfolioMedia`
2. No need to touch individual components — they read from `site.ts`
3. Run `npm run format` to ensure consistency
4. Test locally with `npm run dev`

## Workflow: Creating a Commit

1. Stage changes (`git add` or via VS Code Source Control)
2. Lefthook runs pre-commit: Prettier write + ESLint fix
3. Craft commit message following [Conventional Commits](https://www.conventionalcommits.org/):
   - `feat: add new case study for Project X`
   - `fix: correct broken image fallback`
   - `refactor: simplify contact form validation`
   - Subject ≤100 chars (enforced by commitlint)
4. Commitlint validates message on commit-msg hook
5. Push to trigger CI (lint → type-check → build → test)

## Workflow: Releasing a New Version

Uses [standard-version](https://github.com/conventional-changelog/standard-version) with Conventional Commits:

```bash
npm run release         # Patch: 0.0.1 → 0.0.2
npm run release:minor   # Minor: 0.0.1 → 0.1.0
npm run release:major   # Major: 0.0.1 → 1.0.0
```

This:

1. Bumps `version` in package.json
2. Generates/updates CHANGELOG.md
3. Creates a git tag (`v0.0.2`)
4. Commits the changes

Pushing the tag triggers GitHub Actions to create a GitHub Release automatically.

## Decision: When to Link vs. Embed

This codebase uses a **"Link, don't embed"** principle:

- **Embed** (copy content): Only for critical, task-blocking info (e.g., stack, current commands)
- **Link** (reference): For detailed, reference-heavy docs (Guidelines.md, case study skill, layout guidelines)

Rationale:

- Reduces redundancy and drift
- Keeps agent instructions lightweight
- Directs users to the canonical source of truth

## Common Gotchas

1. **Path alias `@` only works in tests** — Never use in `src/app/*`; use relative imports
2. **Tailwind v4 has no config file** — All styling via utility classes; don't create `tailwind.config.js`
3. **Design tokens in `theme.css` are frozen** — Coordinate with maintainer before modifying
4. **Case studies must be lazy-loaded** — Always use dynamic imports in `CaseStudyPage.tsx`
5. **Contact form is POST, not AJAX** — Redirects to `/contact?sent=1` after submission
6. **Tests exclude `src/app/data/**`\*\* — Mock data or test at component level instead
